# 코드 리뷰 스킬

코드를 일관된 기준으로 검토하고, 발견한 문제를 **심각도 순으로** 정리해 보고한다.

## 1. 리뷰 대상 파악

- 사용자가 특정 파일/디렉토리를 지정하면 그 범위를 리뷰한다.
- 지정이 없으면 최근 변경된 파일을 우선 본다. git 저장소면 `git diff`를,
  아니면 작업 디렉토리의 핵심 소스 파일(예: `index.html`, `*.js`)을 읽는다.

## 2. 리뷰 체크리스트

다음 5개 축으로 코드를 점검한다.

### 🐛 정확성 (Correctness)
- 로직 오류, 경계 조건(빈 배열·0·null), off-by-one
- 비동기/이벤트 처리에서 race condition, 중복 실행
- 이벤트 리스너 누수(붙이고 안 떼는 경우)

### 🔒 보안 (Security)
- 사용자 입력을 그대로 DOM에 넣는 XSS (`innerHTML`)
- 하드코딩된 비밀키, 민감 정보 노출

### 📖 가독성 (Readability)
- 의미 없는 변수명, 과도하게 긴 함수
- 주석과 코드의 불일치
- 매직 넘버(설명 없는 상수)

### 🏗️ 구조 (Structure)
- 중복 코드(DRY 위반), 한 함수가 너무 많은 일을 함
- 관심사 분리(데이터/그리기/이벤트)가 안 된 경우

### ⚡ 성능 (Performance)
- 불필요한 반복 렌더링, 매 호출마다 같은 계산 반복
- 큰 루프 안에서의 DOM 접근

## 3. 보고 형식

발견 항목을 다음 형식으로, **심각도(🔴 High / 🟡 Medium / 🟢 Low)** 순으로 정렬해 보고한다.

```
### 🔴 [심각도] 제목
- 위치: `파일:줄번호`
- 문제: 무엇이 왜 문제인지
- 제안: 어떻게 고치면 되는지 (가능하면 코드 예시)
```

마지막에 **요약**(High N개 / Medium N개 / Low N개)과 **종합 의견**을 1~2문장으로 남긴다.

## 4. 원칙

- 추측이 아니라 **코드를 직접 읽고** 근거를 댄다. 줄 번호를 명시한다.
- 문제가 없으면 "문제 없음"이라고 분명히 말한다. 억지로 만들지 않는다.
- 이 프로젝트는 **학습용**이므로, 왜 문제인지 *이유*를 친절히 설명한다.
